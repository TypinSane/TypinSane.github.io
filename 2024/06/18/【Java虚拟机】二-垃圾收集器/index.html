    # 垃圾收集器

    Serial收集器（一般面向客户端）：最早期的，单线程的，简单的，执行清除工作时需要停止业务线程。优点，简单高效，额外开销小，在硬件受限的环境中应用多（客户端模式下的HotSpot还在应用）(新生代标记复制，老年代标记整理)

    ParNew收集器：Serial收集器的多线程并发版本。

    Parallel Scavenge收集器：新生代，标记复制，并行收集。和ParNew类似，关注的核心点是吞吐量（运行用户代码时间/（运行用户代码时间+垃圾收集时间））

    -XX:MaxGCPauseMillis 大于0，GC时间（会影响GC的吞吐量和新生代的空间，印象GC的频率）

    -XX:GCTimeRatio 垃圾收集时间占总时间的比例。

    Parallel Old收集器：标记整理，

    CMS收集器：关注的核心是最短回收停顿时间 。提高响应速度。缺点是对CPU资源敏感、造成内存碎片、无法处理浮动垃圾（GC过程中用户线程产生的垃圾）。

    Garbage First收集器：

    ​ G1改变了对内存布局的方式，不再是直接将内存区域划分为Eden、Survivor、Old等，因为将内存划分为大块的区域，在进行垃圾收集时必然需要对整块区域进行回收，势必会有较长的停顿，以及更高的回收代价。

    ​ 于是就将内存区域划分成固定尺寸的更小的内存单元（Region）。内存单元之间通过逻辑的方式进行关联，以及扮演分代收集的不同内存区域的角色。似乎是一种模块化的思想。

    ​ -XX:G1HeapRegionSize ：1MB～32MB，设定每个Region的大小，如果超过一个region容量的对象，则会放在多个连续的Region中构成一个HumongousRegion。

    ​ G1的核心在于，收集器会判断每一块区域回收的价值（空间大小、所需时间）维护一个列表，根据用户要求的停顿时间（-XX:MaxGCPauseMillis 默认200ms), 优先处理回收价值收益最大的Region

    ​ G1的难点：

    1. 如何来理清楚处在不同Region中对象的关联关系？在每个region中都维护有一个key-value的结构，用来记录当前的region中的对象指向哪个region？哪个region中的对象又指向了当前这个对象，所以G1收集器至少要多消耗10-20%的内存来维持工作。
